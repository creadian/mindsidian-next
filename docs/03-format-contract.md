## 1. The grammar

A Mindsidian file has four zones, in order:

```
[frontmatter]      YAML between --- fences, only at byte 0
[preamble]         anything between frontmatter and the first H1 (rare; usually empty)
# Root title       exactly one H1 = the root node
[body]             headings + tab-indented bullet lists = the tree
```

### 1.1 Frontmatter

| Key | Type | Meaning | Written by |
|---|---|---|---|
| `mindmap-plugin: basic` | string | Marks the file as a mindmap. The view only activates when this key is present. Never remove it. | created once (`basicFrontmatter` in `constants.ts`) |
| `mindmap-zoom: 78` | int 20–300 | Last zoom level (%). Read on open (fallback: plugin `defaultZoom`, clamp to 20–300, round). Written via `processFrontMatter` on view close/unload. | view close |

Rules:

- **F1.** Frontmatter is recognized **only** when the file's first non-empty content is a `---` line.
  (v0.5.47 uses `FRONT_MATTER_REGEX = /^(---)$.+?^(---)$.+?/ims` on the *trimmed whole file* — it can match
  `---` horizontal rules mid-document in a file without frontmatter and **delete everything between them**.
  v2 MUST anchor frontmatter detection to the top of the file. This is a deliberate, documented compat *fix*.)
- **F2.** All frontmatter keys v2 does not manage (tags, date, aliases, anything) are preserved **byte-verbatim**.
  v0.5.47 does this by slicing the original frontmatter substring (`getFrontMatter()`, using `frontmatterPosition.end.offset`)
  and prepending it to the regenerated body. v2 must do at least as well: never re-serialize YAML it didn't change.
  The only structured write is `mindmap-zoom` (and it must not reorder/reformat other keys — use Obsidian's `processFrontMatter`).
- **F3.** Frontmatter with blank lines inside (`---\n\nmindmap-plugin: basic\n\n---`) is valid and occurs in real
  files (`Test-mindmap-broken.md`, `Praxis workstreams.md`). Must parse.
- **F4.** Saved layout: `frontmatter + "\n\n" + body` (v0.5.47: `yamlString = frontmatter-slice + '\n\n'`).

### 1.2 Root

- Exactly one root node. The **first H1** (`# Title`) is the root.
- Empty file / no H1: v0.5.47 synthesizes a root from the file basename. v2: same (synthesize, but **do not write
  to disk until the user actually edits** — see E12).
- Multiple H1s: markmap puts them under an invisible root; on save v0.5.47 keeps only the traversal of the single
  root tree. v2: treat the first H1 as root and **attach later H1s as children of root** (no content drop).

### 1.3 Heading vs. list item — the `headLevel` rule

Setting: `headLevel` (UI: "Max level of node to create a heading"), default **2**, range 1–6.
Node depth: root = level 0, its children = level 1, etc. (`getLevel()` in `INode.ts`).

Serialization (`getMarkdown()`, `mindmap.ts:3407`):

- **Level < headLevel** → ATX heading with `level+1` hashes, preceded by one blank line:
  level 0 → `# `, level 1 → `## `, level 2 → `### ` (only if headLevel ≥ 3), …
- **Level ≥ headLevel** → list item: `"\t" × (level − headLevel)` + `- ` + text.
  **Indentation is TABS, one tab per depth step, bullets always `- `** (never `*`/`+`, never numbered).
- With default headLevel=2, the canonical shape is: `# root`, `## section`, then bullets — exactly what every
  real file shows.

Parsing accepts much more than it emits (see §1.8 normalization), but the **canonical output** is exactly the above.

### 1.4 Block IDs (`^xxxxxxxx-xxxx-xxxx`) — node identity + fold state

- Syntax: node line ends with `" ^" + id`, id = three hex groups `8-4-4` lowercase, e.g. `^6499845f-ba31-8078`
  (generated by `uuid()` in `MindMapView.ts:23` — `S4()+S4()+'-'+S4()+'-'+S4()`).
- **Semantics in v0.5.47:** a trailing ID means *this node is collapsed (folded)*. Parse: regex
  `/^.+ \^([a-z0-9\-]+)$/` → `id` adopted as the node's id, `" ^id"` stripped from display text,
  `expanded = false`. Serialize: ID is appended **only to collapsed nodes**, and **only when the setting
  `foldStatePersistence === 'markdown'`** (the default). Expanded nodes get a fresh random id in memory each
  load and write no marker.
- Works on both headings (`## Workboard ^6499845f-ba31-8078`) and bullets (`- test ^77fa756c-1db3-6c5c`).
- **v2 rules:**
  - **B1.** A trailing `^id` matching the strict 8-4-4 hex shape → fold marker: collapsed + stable id. Preserve the
    exact id through the roundtrip while the node stays collapsed.
  - **B2.** Any other trailing `^block-id` (Obsidian user block references like `^quote1`, `^my-anchor`) is **NOT**
    a fold marker: keep it verbatim as part of the node text, node stays expanded. (v0.5.47's loose
    `[a-z0-9\-]+` regex swallows genuine Obsidian block refs and force-collapses those nodes — documented compat fix.
    Consequence to verify in tests: a user block ref must survive save so links to it don't break.)
  - **B3.** Expanding a collapsed node removes its marker on next save; collapsing adds one. This is the one
    *legitimate* diff a fold action produces.

### 1.5 Fold state / zoom / highlights / checkboxes — persistence summary

| Feature | Persisted as | Notes |
|---|---|---|
| Fold state | mode `markdown` (default): trailing `^id` on collapsed nodes. Mode `plugin-data`: text-paths (`"Root > A > B"`) in plugin `data.json`, markdown stays clean. Mode `none`: not persisted. | `settings.foldStatePersistence`; `applyCollapsedPaths` / `collectCollapsedPaths` in `MindMapView.ts` |
| Zoom | `mindmap-zoom` frontmatter int | written on close, clamp 20–300 |
| Highlight | whole-node wrap `<mark style="background:#A7E8A4;">text</mark>` stored inline in the node text | `applyHighlight` in `INode.ts:404`; regex tolerant: `^<mark\s+style="[^"]*">…</mark>$`. Real palette colors seen in files: `#A7E8A4`, `#A4C9F7`, `#CDA8E6`, `#FFD580` |
| Checkbox / task | bullet text prefix `[ ] ` (todo) / `[x] ` (done), i.e. the line reads `- [ ] foo` / `- [x] foo` | parse accepts `[X]` too; **bullets only — headings never carry a checkbox** and a task node promoted above headLevel silently loses its prefix in v0.5.47 (v2: same rule, but warn/avoid promoting task nodes) |

### 1.6 Node text content — what may appear inside a node

Node text is a **single line of Obsidian markdown**, rendered by `MarkdownRenderer`. The serializer stores it
verbatim (after `.trim()`); the parser strips only the task prefix and the fold `^id`. Everything else passes through:

- **Wikilinks** `[[Note]]`, `[[Note#Heading]]`, `[[Note|display]]` — verbatim. (Inside tables Obsidian needs
  `\|`; mindmap files don't use tables, so no escaping is applied by the plugin.)
- **Inline code** `` `like this` `` — verbatim (Praxis file uses whole-bullet inline-code quotes).
- **Bold/italic/strikethrough/highlight** `**b**`, `_i_`, `==hl==`, `<mark …>` — verbatim.
- **Raw URLs, embeds** `![[image.png]]`, math, emoji — verbatim.
- **Pipes** `|` in text — verbatim (no table semantics inside a node).
- **Multi-line text** (contains `\n`, e.g. via `<br>`-less line breaks in edit mode):
  v0.5.47 **splits each line into its own sibling bullet** on save (first line keeps the task prefix) —
  i.e. a multi-line node does not roundtrip; it becomes N nodes. v2: same serialization is acceptable,
  but it must never *drop* a line. Exception — **code fences**: text starting with ``` ``` ``` is emitted as an
  empty bullet `- ` followed by the fence lines indented two extra spaces, wrapped in blank lines
  (`mindmap.ts:3447`). v0.5.47 `.trim()`s each fence line (destroys code indentation — see E10).
- Empty text: rendered as placeholder "Sub title"; an empty node serializes as a bare `-` line.
  (Note: real files are full of literal "Sub title" — the placeholder *string* gets persisted once the node is
  saved. v2 keeps this behavior: placeholder text is real text.)

### 1.7 Blockquote-style child text

`> quoted line` under a bullet: markmap parses a blockquote node; v0.5.47's `transformData` takes the
blockquote's **first child only**, prefixes the text with `"> "`, and **drops further children/paragraphs**
(`flag = false`, `MindMapView.ts:1168`). On save it becomes `- > quoted line`. v2: a blockquote becomes one node
per quote line (no drop), text keeps its `> ` prefix verbatim.

### 1.8 Input normalization (parse-time "assimilation")

`normalizeBullets` (`MindMapView.ts:1034`) implements the v0.5.47 principle: *assimilate everything into the
tree; never silently drop content*. v2 must honor the same outcomes:

- **Headings** require `#` + space (`/^#{1,6}\s/`). `#tag` and `###no-space` are NOT headings → become bullet text.
- **Bare text** (no bullet, no heading) → becomes a bullet at its indentation.
- **Indentation**: tabs count 1 each; spaces are converted using the smallest space-indent found in the file
  (default 2). Output is always tabs.
- **Orphaned indentation** (deeper than parent+1) is clamped to parent+1. First bullet after a heading is forced
  to depth 0 under that heading.
- **Chained bullets** `- - X` collapse to `- X` (avoids empty intermediate nodes).
- **Horizontal rules** `---` / `***` / `___` → become a bullet with that literal text (`- ---`).
- **Numbered lists** `1. item` → accepted as list items; serialize back as plain `- ` bullets
  (ordering implied by sibling order; the `1.` marker itself is not preserved as a marker).
- **Code fences** pass through normalization untouched (state-tracked).
- **Blockquote lines** pass through to the markmap parser (then §1.7 applies).

## 2. PRESERVATION RULE (overrides everything)

> **Any markdown construct the parser does not understand must be PRESERVED, never dropped.**

Concretely for v2:

- **P1.** If a line cannot be classified (heading / bullet / quote / fence / blank), it is *assimilated* as a
  node whose text is the line verbatim (v0.5.47 behavior) — never discarded.
- **P2.** Frontmatter keys, preamble text before the first H1, and trailing unknown content must survive a
  save byte-for-byte.
- **P3.** Inside node text, no character-level rewriting except: leading/trailing whitespace trim, task-prefix
  extraction, fold-`^id` extraction.
- **P4.** If parsing *fails* (exception, malformed structure beyond repair), v2 must **refuse to save** rather
  than write a partial tree over the file. (v0.5.47 has no such guard; this is a v2 hardening requirement.)
- **P5.** A structural normalization (e.g. spaces→tabs, bare text→bullet) is allowed **only** if it is listed in
  §1.8 / §3 as a *documented normalization*; anything else byte-identical.

## 5. Canonical serialization reference (v2 emitter)

For a node at `level` with `headLevel = H`:

```
level == 0          →  "# " + text [+ " ^id" if collapsed]          (preceded by frontmatter + blank line)
0 < level < H       →  "\n" + "#"*(level+1) + " " + text [+ " ^id" if collapsed]
level >= H, normal  →  "\t"*(level-H) + "- " + [task prefix] + text [+ " ^id" if collapsed]
level >= H, empty   →  "\t"*(level-H) + "-"
level >= H, fence   →  blank line, "\t"*(level-H) + "-", fence lines at +2 spaces (byte-exact), blank line
```

Children follow their parent depth-first in document order. Output ends without trailing whitespace
(`md.trim()`), then `frontmatter + "\n\n" + body` is written.

Task prefix: `"[ ] "` or `"[x] "`. Fold id: only in `markdown` persistence mode, only on collapsed nodes,
strict `^hhhhhhhh-hhhh-hhhh` lowercase hex.
